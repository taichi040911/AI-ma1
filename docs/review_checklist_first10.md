
# 最初の10チケット専用のレビュー観点チェックリスト

## 使い方
- 1チケット完了ごとに、このチェックリストでレビューする
- **仕様 / 実装 / テスト / UX / 次チケット影響** の順で確認する
- 1つでも不明点があれば、Codex に差し戻す
- 可能なら、各項目を `OK / NG / 要確認` で記録する

---

# 共通チェック（全10チケットで毎回見る）

## 1. 仕様整合
- 対象チケットIDが明確か
- 対象APIまたは画面名が明確か
- `AGENTS.md` のルールに反していないか
- `integrated_openapi_final.yaml` と矛盾していないか
- `final_screen_list_with_billing.md` / `screen_api_mapping.xlsx` と矛盾していないか
- 不明点があった場合、仮定が明記されているか

## 2. 実装品質
- 変更ファイル一覧が提示されているか
- 変更範囲が過大ではないか
- 既存機能を壊す破壊的変更が入っていないか
- 命名が分かりやすいか
- ハードコードが不要に増えていないか
- エラー処理があるか

## 3. テスト
- 少なくとも1つ以上テストが追加されているか
- 正常系があるか
- 異常系または境界値があるか
- lint / test 実行結果が共有されているか
- テスト名から何を確認しているか分かるか

## 4. UX / 画面
- loading state があるか
- empty state があるか
- error state があるか
- 戻る導線があるか
- CTA が分かりやすいか
- 文言が煽りすぎていないか

## 5. 次チケット影響
- 次のチケットが着手しやすい状態か
- OpenAPI 更新が必要なら反映済みか
- migration が必要なら追加済みか
- 次チケットへの注意点が書かれているか

---

# 1. AUTH-001 新規登録API レビュー観点

## 必須
- `POST /auth/register` が実装されている
- メール重複時に適切なエラーが返る
- 必須項目不足時に適切なエラーが返る
- 成功時に `user_id` など最低限の情報が返る
- 規約同意のチェックがある

## 見るポイント
- パスワードの扱いが安全か
- レスポンスが OpenAPI と一致しているか
- 失敗時のエラー文言が曖昧すぎないか

## Done 判定
- 正常登録できる
- 重複エラーが返る
- 必須不足で弾かれる

---

# 2. AUTH-003 OTP送信API レビュー観点

## 必須
- `POST /auth/send-otp` が実装されている
- 送信先の形式チェックがある
- `expires_in_seconds` が返る
- 再送制限またはレート制限がある
- 送信失敗時のエラーがある

## 見るポイント
- 再送の間隔が極端でないか
- 送信ログや監査観点が最低限あるか
- 本番用の送信実装とモックの切替方針が分かるか

## Done 判定
- 送信成功
- レート制限あり
- 期限情報あり
